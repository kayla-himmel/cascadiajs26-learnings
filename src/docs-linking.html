Better Documentation Linking

TSDoc's built-in tags let you link between functions in VS Code hover tooltips. The custom @ref convention takes it a step further by making those links machine-verifiable — so broken references get caught in CI before they mislead anyone.

What TSDoc gives you natively

TypeScript's JSDoc support includes two tags for cross-referencing symbols:

  • {@link} — renders as a clickable go-to-definition link inside VS Code hover tooltips. Ctrl/Cmd-click jumps straight to the target.
  • @see — adds a "See also" section in the JSDoc comment block. Useful for grouping related functions or pointing to docs.

Neither tag validates that the linked target actually exists. Rename a function or move a file and the link silently becomes stale. 

/**
 * Applies the color scheme.
 * @see {@link getColorScheme} for reading the current scheme
 */
export function applyColorScheme(scheme: ColorScheme): void { ... }

The @ref convention

@ref is a custom inline annotation — not a TSDoc standard — used in this repo to declare which documentation files a source file depends on. Unlike {@link}, these can be validated automatically. 

// @ref docs/light-dark.md#usage [describes] — light-dark() theme tokens
// @ref docs/better-docs-linking.md#ref-convention [meta] — explains this annotation format

Each @ref line has four parts:

  • path File path relative to the repo root — e.g. docs/light-dark.md
  • #anchor Optional. Must match an ## heading in the target file exactly.
  • [relationship] A human-readable label: [describes], [meta], [implements], etc.
  • — note Optional freeform comment explaining the connection. Ignored by the validator.

See it in action: example.ts

Below are the key annotated sections from src/ts/example.ts. The file-level @ref comments at the top declare which docs this module relates to. The JSDoc blocks show @see and {@link} working together on individual functions. 

// @ref docs/better-docs-linking.md#ref-convention [meta] — this file uses the @ref convention
// @ref docs/light-dark.md#how-it-works [describes] — this module manages the light-dark() color scheme

/**
 * The color scheme the user has selected or that the system is using.
 *
 * @see {@link applyColorScheme} to set this value programmatically
 * @see {@link getColorScheme} to read the current value
 */
export type ColorScheme = 'light' | 'dark' | 'system';

// @ref docs/light-dark.md#usage [describes] — this function drives the light-dark() switch
/**
 * Sets the active color scheme by writing `data-theme` onto the `<html>` element.
 *
 * @param scheme — The {@link ColorScheme} to apply.
 *
 * @see {@link getColorScheme} to read the current scheme before toggling
 *
 * @example
 * ```ts
 * applyColorScheme('dark');   // forces dark mode
 * applyColorScheme('system'); // reverts to OS preference
 * ```
 */
export function applyColorScheme(scheme: ColorScheme): void { ... }

Try it in VS Code

Open src/ts/example.ts and hover over a function name to see the @see link rendered in the tooltip. Ctrl/Cmd-click a {@link} reference to jump straight to the target symbol.

Validating with check-refs.mjs

check-refs.mjs scans every .ts file under src/ts/, extracts each @ref annotation, and verifies that:

  • The referenced file exists in the repo
  • If an #anchor is specified, the matching ## heading exists in that file

Run it with:

npm run check:refs

Example output (all passing):

Checking @ref annotations in src/ts/...

src/ts/example.ts:15         @ref docs/better-docs-linking.md#ref-convention ✅
src/ts/example.ts:16         @ref docs/light-dark.md#how-it-works            ✅
src/ts/example.ts:63         @ref docs/light-dark.md#usage                   ✅
src/ts/example.ts:90         @ref docs/color-mix.md#usage                    ✅
src/ts/example.ts:116        @ref docs/contrast-color.md#overview            ✅

5 refs checked, 0 failed. All good!

The script exits with code 0 when all refs pass and code 1 when any fail — so a broken reference stops the CI build.

Break it on purpose

Open src/ts/example.ts and change one @ref path to docs/nonexistent.md, then run npm run check:refs. The failing ref will be flagged with a clear reason and the script will exit 1.

Automated with GitHub Actions

The .github/workflows/refs.yml workflow runs check-refs.mjs on every push and pull request. No manual step needed — the CI job catches broken refs before they merge. 

name: Check @ref annotations

on:
  push:
    branches: ["**"]
  pull_request:
    branches: ["**"]

jobs:
  check-refs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - name: Validate @ref annotations
        run: node check-refs.mjs
